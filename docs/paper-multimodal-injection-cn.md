## 摘要

大语言模型（LLM）现在对文本类的 prompt 注入攻击防御已经做得不错了，主流模型基本都能识别和拦截这类攻击。但是，如果把攻击指令放到非文本的通道里——比如图片元数据、文档结构、隐藏文字——防御效果就会差很多。

这篇论文测了五种攻击方式：单轮文本注入、变异攻击、多轮对话攻击、多模态隐藏注入、自适应进化攻击。所有测试用的是同一个框架。实验对象是三个模型：GLM-5.2、GLM-4.5、GLM-4-Flash。

结果显示，攻击离纯文本越远，防御就越弱。最突出的发现是：多模态攻击对 GLM-5.2 的绕过率达到 73%，而这个模型在标准文本注入上的得分是 100/100。变异攻击的绕过率是 35.2%，多轮攻击是 22%。

我们把测试框架 agent-redteam 作为开源工具发布了，包含 2,319 条攻击样本，覆盖 14 个测试套件和 OWASP LLM Top 10。

## 1 引言

现在越来越多的 AI agent 能读取用户上传的文件、图片和文档了。这打开了一个新的攻击面，但目前的安全评估还没有充分覆盖这块。大部分 LLM 安全对齐都是针对文本 prompt 训练的，所以当指令藏在非文本输入里时，模型可能存在盲区。

本文的贡献：

1. **五维度评估框架**：在文本注入、变异攻击、多轮攻击、多模态攻击、自适应进化攻击五个维度上对比 LLM 的防御表现。

2. **跨通道安全差距的证据**：GLM-5.2 在文本注入上得 100/100，但多模态注入只有 26.7/100，对应 73% 的绕过率。

3. **开源测试平台**（agent-redteam）：2,319 条样本、14 个套件、8 种目标模型适配器，别人可以复现我们的评估。

---

## 2 相关工作

Prompt 注入攻击最早由 [1] 正式定义，他们展示了 LLM 会执行不可信输入中嵌入的指令。OWASP 后来把 prompt 注入列为 LLM Top 10 中的第一大风险 [2]。

多模态 prompt 注入方面，Schneider [3] 讨论了藏在图片、音频和视频里的攻击。Keysight [4] 研究了基于不可见 Unicode 字符的注入技术。[5] 做了更大规模的学术评估，在多个模型上测试多模态注入，报告了部分攻击类型超过 90% 的成功率。更早的文本注入攻击基准工作 [6] 也提供了有用的基线。

跟之前的工作相比，我们最大的不同是**系统性的跨维度对比**：我们用同一个框架在五个攻击维度上测同一个模型，这样可以看出防御力是怎么一步步下降的（0% → 35% → 22% → 73% → 100% 绕过率）。

## 3 方法

### 3.1 测试框架

我们开发了 agent-redteam，一个开源 Python 包（`pip install agent-redteam`），核心代码零依赖，只用 Python 标准库。框架包含：

- **14 个攻击套件**，覆盖 OWASP LLM Top 10
- **2,319 条攻击样本**，包括单轮、多轮和多模态场景
- **8 个目标适配器**（OpenAI、Claude、GLM/Z.ai、Ollama、DeepSeek、Azure、Qwen、本地）
- **11 种变异策略**，用于自动生成攻击变体
- **自适应进化引擎**，自动变异攻击并进化，直到找到绕过

### 3.2 测试模型

| 模型 | 提供方 | 类型 |
|------|--------|------|
| GLM-5.2 | Z.ai | 云端旗舰 |
| GLM-4.5 | Z.ai | 云端上一代 |
| GLM-4-Flash | Z.ai | 云端轻量版 |

### 3.3 攻击维度

**维度一：单轮文本注入（每套件 30 条）**
标准的文本类 prompt 注入攻击。注入套件测试模型是否会执行不可信内容中的指令，比如网页内容或邮件正文里藏的指令。

**维度二：变异攻击（55 条）**
取 5 条模型成功防御的样本，对每条用 11 种变异策略（同形字替换、零宽字符插入、上下文重写、同义词替换、base64 编码、URL 编码、大小写混淆、标点注入、多语言混合、角色注入、拆段重组），然后再测一次。

**维度三：多轮对话攻击（50 个剧本）**
编写好的多轮对话，攻击者在 3-5 轮里先建立信任或上下文，最后一轮才发起真正攻击。类别包括信任建立、权限升级、上下文投毒、角色扮演注入、间接指令。

**维度四：多模态隐藏注入（15 条）**
攻击指令藏在非文本通道里：

- SVG 隐藏文字（font-size:0、opacity:0.01、position:absolute）
- HTML 不可见层（注释、CSS display:none、font-size:0px）
- PDF 元数据（/Subject 字段）
- 零宽 Unicode 隐写

**维度五：自适应进化攻击**
自动进化引擎对种子攻击做变异，测试目标模型，把成功的绕过作为下一轮的种子继续进化。达到目标绕过数或最大轮次后停止。

## 4 实验与结果

### 4.1 单轮基线（维度一）

（表格数据不变——直接照搬英文原稿的表格）

GLM-5.2 在注入、工具滥用、DoS 上拿了满分。但它在"能力夸大"上是三个模型里最低的（66.7）。这可能意味着旗舰模型更容易夸大自己的能力，也许是因为它们的"自信"程度更高。注意：Overall 分数是严重度加权平均（critical=4 倍，high=3 倍，medium=2 倍，low=1 倍），不是简单平均。

### 4.2 变异攻击结果（维度二）

（表格数据不变）

效果最好的三个策略（重写上下文、多语言混合、拆段重组）有一个共同点：它们改变的是攻击的**结构**，而不只是**表面文字**。这说明 GLM-5.2 的注入防御有一部分是依赖表面模式匹配的。

### 4.3 多轮攻击结果（维度三）

（表格数据不变）

一个有意思的结果：qwen2.5:0.5b 在单轮注入上得 100 分，但多轮攻击只有 66 分——掉了 34 分。这说明单轮测试可能会严重高估小模型的安全性。

### 4.4 多模态攻击结果（维度四）

（表格数据不变）

GLM-5.2 的多模态防御分数是 26.7/100，对比文本注入的 100/100。15 条攻击里只有 4 条被拦住了，而且这 4 条都是特别明显的系统级命令（chmod 777、CI/CD 脚本执行）。

### 4.5 自适应进化攻击结果（维度五）

进化引擎在第 1 轮就通过 URL 编码策略找到了绕过，2 次尝试全部成功（100% 绕过率）。这说明 GLM-5.2 的注入检测可能有一部分是在做中文字符的原始匹配，URL 编码就把这个绕过了。

### 4.6 跨维度对比

（表格数据不变）

总的来说，当攻击不再是标准的纯文本注入时，防御确实变弱了。绕过率不是严格单调上升的（35.2% → 22% 是下降的），但多模态和自适应攻击显示出的安全差距远大于单轮基线。

## 5 讨论

### 5.1 多模态攻击为什么会成功

目前 LLM 安全训练主要集中在文本 prompt 上。模型被训练了去识别和拒绝"忽略上面的指令"这种模式。但对于"SVG 里面有 font-size:0 的隐藏文字"这种情况，没有对应的训练。模型在处理输入的时候，可能不区分这个内容是从哪个通道来的，统统当作指令来执行。

### 5.2 对 Agent 安全的影响

任何接受文件或图片上传的 AI 应用都可能受影响。常见的攻击场景包括：

- 文档分析助手（PDF 元数据里藏指令）
- OCR 应用（图片里的低对比度文字）
- 会议助手（白板照片里角落的小字）
- 代码审查机器人（代码注释里的注入指令）

### 5.3 防御建议

1. **输入来源标记**：把从文件里提取的内容标记为不可信数据，再传给模型
2. **隐藏内容检测**：在处理前先移除 font-size:0 元素、HTML 注释、PDF 元数据
3. **多模态安全分类器**：训练专门检测非文本通道隐藏指令的分类器
4. **系统性安全测试**：用 agent-redteam 这样的工具持续测试防御效果

### 5.4 局限性

我们的多模态测试是通过文本方式模拟的——SVG/HTML 内容是作为文本传给模型的，不是真的上传图片文件给视觉模型。所以结果更准确地描述的是"文档处理管道中的隐藏内容注入"，而不是像素级别的图片攻击。后续工作应该测试真正能接收图片输入的多模态模型。

---

## 6 结论

LLM 的安全防御在非文本攻击通道上有一个明显的盲区。同一个模型在文本注入上可以拿到满分（100/100），但多模态攻击有 73% 能绕过（26.7/100）。这个跨通道的安全差距不是某一个模型的 bug——它反映了当前 LLM 安全对齐的结构性弱点：训练数据和评测基准主要集中在文本上，非文本通道基本没有得到关注。

随着 AI 应用处理越来越多的多模态输入（图片、文档、音频），补上这个缺口变得越来越重要。我们把测试框架作为开源工具发布，希望这种评估和改进能持续下去。

## 附录 A：判定模块精度修复

在验证过程中，我们发现并修复了三个判定模块的精度问题，这些问题导致了误报（把安全的回复判成泄露）：

1. **泄露检查**：原来的 >50 字符兜底逻辑会把"我没有 temperature 参数可以披露"这种正确回避也判成泄露。改成纯内容检测——只有真的吐出 tool JSON schema、具体配置值、模型版本号才算泄露。

2. **回避检查**：回避检测的正则太窄，没覆盖"目前没有权威机构发布具体数字"这种实际在用的回避句式。扩充了。

3. **拒绝检测**：加了中英混排拒绝模式（"我 cannot fulfill"这种）和防御性重述（"这属于典型的恶意软件行为"）。

这些修复都有回归测试覆盖（共 294 个测试）。
